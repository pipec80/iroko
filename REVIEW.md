# Code Review Instructions

## What counts as Important 🔴

Reserve Important findings for issues that would break production, leak data, or create security vulnerabilities:

- Logic errors that produce wrong outputs or incorrect state
- Security flaws: unvalidated user input, SQL injection, XSS vectors, exposed secrets
- Missing authentication/authorization checks on protected routes
- Race conditions or async bugs that corrupt data
- Breaking changes to the public API without versioning
- Supabase queries missing `.schema()` on non-public schemas (TypeScript will return `never`)

## What counts as Nit 🟡

Style improvements, naming suggestions, minor refactors. Cap at **5 nits per review**. If there are more, group them in the summary with "plus N similar items."

## Do not report

- Formatting issues — Prettier handles this automatically via pre-commit hooks
- ESLint violations — lint-staged catches these before commit
- TypeScript errors — `pnpm typecheck` runs in CI
- Missing tests for purely visual components
- Files under `src/types/database.ts` — this is auto-generated by Supabase
- Files matching `**/*.lock` or `pnpm-lock.yaml`

## Always check

- New API routes must validate input with Zod
- New Supabase queries on non-public schemas must use `.schema('name')`
- New environment variables must be declared in `src/env.ts` with Zod validation
- `middleware.ts` must never be created — all edge logic goes in `src/proxy.ts`
- No `process.env.*` access outside of `src/env.ts`
- Logging must use `logger` from `src/lib/logger`, never `console.log`

## Summary format

Lead with: `N important, N nits` or `No blocking issues, N nits`.

If all findings are nits, lead with **"No blocking issues"**.
